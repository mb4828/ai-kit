name: memory-management
description: Maintain the vault's two-tier shared memory so agents understand Matt's people, projects, shorthand, and preferences across sessions. Use when asked to remember, recall, clarify, add, update, promote, demote, or clean up durable context.
---

Use `Cowork/memory/MEMORY.md` as the hot, compact index and `Cowork/memory/` as deep storage. Do not use provider-managed memory as the vault's source of truth.

1. Read `Cowork/memory/MEMORY.md` first. Follow only its relevant links; search the wider memory tree before asking about an unfamiliar person, project, acronym, or preference.
2. Decode user shorthand in layers: the index and linked context first, then `glossary.md`, then the applicable `people/`, `projects/`, or `context/` file. Ask Matt when the meaning is still unknown.
3. Store acronyms, nicknames, aliases, codenames, and internal terminology in `glossary.md`. Store rich person or project detail in dedicated files; store broader life or workflow context under `context/`.
4. Add a new deep-memory file to `MEMORY.md`. Promote only frequently useful active facts into the index; demote stale or historical detail from the index without deleting its deep record.
5. Record only durable, supported facts. Treat therapy, journal, meetings, and job-search data as confidential; do not quote or broadly summarize it unless Matt asks.
6. When a person or organization also needs an Obsidian graph node, use `manage-entities`; keep rich context in memory rather than the entity note.

Keep the index concise and scannable. Capture alternate names because they are essential for reliable recall.
