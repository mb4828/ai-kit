# React + TypeScript Testing Standards

Apply these standards when writing or reviewing frontend tests. Stack: **Vitest** + **React Testing Library** + `@testing-library/jest-dom` + `@testing-library/user-event`, running in jsdom.

## Core Principles

- **Test behavior a user can observe**, not implementation details. Query the rendered output the way a user or assistive tech would.
- **Tests must fail when the code is broken.** Coverage without meaningful assertions is worthless.
- **Fixtures must match the real data contract.** A hand-built fixture that reflects your assumptions rather than what the backend emits will hide schema drift instead of catching it.

## Coverage Requirements

**Minimum 80% on lines, functions, branches, and statements** for `src/`, enforced in `vite.config.ts` so a regression fails the build:

```typescript
test: {
  environment: "jsdom",
  globals: true,
  setupFiles: ["./vitest.setup.ts"],
  coverage: {
    provider: "v8",
    include: ["src/**/*.{ts,tsx}"],
    exclude: ["src/main.tsx", "src/**/*.test.{ts,tsx}"],
    thresholds: { lines: 80, functions: 80, branches: 80, statements: 80 },
  },
}
```

```bash
yarn test           # vitest run
yarn test:coverage  # vitest run --coverage (enforces thresholds)
```

Exclude only the app entry point (`main.tsx`) and test files themselves. Don't chase 100% — cover real logic and user-visible branches, not every defensive line.

## Query by Behavior, Not Structure

Prefer queries in this order: **accessible role/name → label text → visible text → test id** (last resort, for icon-only elements with no accessible text).

```tsx
// GOOD - the user finds controls by their accessible name
await user.click(screen.getByRole("button", { name: "Play (Space)" }));

// GOOD - a passenger icon exposes its data through an aria-label
expect(screen.getByLabelText("#1 to 5 [-]")).toBeInTheDocument();

// Acceptable only when there is no accessible text (MUI icon buttons):
expect(screen.getByTestId("PlayArrowIcon").closest("button")).toBeDisabled();
```

Assert on what the user sees. Never reach into component internals, state, or private fields.

## Test Real Code Paths; Mock Only Heavy Boundaries

Drive real logic end-to-end. Mock only expensive or environment-specific dependencies — a charting library that needs layout, a network call — never the unit under test.

```tsx
// GOOD - stub only the heavy chart, then assert on the dataset it receives
let lastProps: { dataset?: Array<Record<string, unknown>> } | undefined;
vi.mock("@mui/x-charts/BarChart", () => ({
  BarChart: (props) => { lastProps = props; return <div data-testid="bar-chart" />; },
}));
```

Exercise real flows where practical: at the `App` level, drive an actual `FileReader` by firing a `change` event with a real `File`, and assert the resulting views render. That kind of integration test is exactly what catches contract drift a unit test misses.

## Fixtures Must Reflect the True Schema

The most dangerous test data is a fixture that looks right but doesn't match production output — it makes both the code and the test agree on a lie.

- Build fixtures from the **actual emitted shape**. Type them with the real domain type (`const elevator: FrameElevator = {...}`) so `tsc` rejects a fixture that drifts from the schema.
- For contract-sensitive parsing, prefer a **real sample generated by the backend** committed as a fixture, plus a round-trip parse test. A single such test turns any future schema change into a red build.
- Factor shared fixture construction into typed factory helpers (`makeElevator`, `makePassenger`, `sampleOutputFile`) — the frontend analogue of a `conftest.py`. Give each an `overrides` parameter for per-test variation.

```tsx
function makeElevator(overrides: Partial<FrameElevator> = {}): FrameElevator {
  return { id: 1, floor: 2, direction: "idle", phase: "idle", passenger_count: 0, target_floor: null, ...overrides };
}
```

## Testing Hooks

Test custom hooks directly with `renderHook`, wrapping every state change in `act`.

```tsx
const { result } = renderHook(() => usePlayback(2, true));
act(() => result.current.play(1));
```

**Use fake timers for intervals and animations.** Anything driven by `setInterval`/`setTimeout` (playback auto-advance, enter/exit transitions) must be tested deterministically:

```tsx
beforeEach(() => vi.useFakeTimers());
afterEach(() => vi.useRealTimers());

it("advances one tick per second and pauses at the end", () => {
  const { result } = renderHook(() => usePlayback(2, true));
  act(() => result.current.play(1));
  act(() => vi.advanceTimersByTime(1000));
  expect(result.current.tick).toBe(1);
  act(() => vi.advanceTimersByTime(1000));
  expect(result.current.tick).toBe(2);
  expect(result.current.playbackRate).toBeNull(); // auto-paused at the end
});
```

Cover the full state machine: initial state, each transition, boundary clamping, disabled mode, and reset.

## Test Coverage Scenarios

For every unit, cover:

1. **Happy path** — valid input renders/returns the expected result.
2. **Edge cases** — empty timeline, single item, start/end boundaries, unknown ids skipped.
3. **Error conditions** — invalid input surfaces a clear, user-visible error (parse failure shows an alert; a corrupt later load keeps the current view and shows the error).
4. **Disabled / guard branches** — keyboard handler ignores events from editable fields and when disabled; controls disabled at timeline ends.

## Test Structure

- **Colocate** tests with source: `usePlayback.ts` → `usePlayback.test.ts`, `StatsPanel.tsx` → `StatsPanel.test.tsx`.
- **Group** related tests in a `describe`; name each `it` for the behavior it asserts, phrased as a user-visible outcome.
- **Isolate** — reset shared spies/handlers in `beforeEach`; no shared mutable state leaking between tests. Restore real timers in `afterEach`.
- Keep unit/component tests fast (jsdom, no real network or disk).

## Quality Checklist

Before submitting frontend tests, verify:

- [ ] `yarn test:coverage` passes with all four thresholds ≥ 80%
- [ ] Queries use role/label/text; test ids only for icon-only elements
- [ ] Assertions are specific and fail when the code breaks (verify by breaking it)
- [ ] Real code paths exercised; only heavy boundaries (charts, network) mocked
- [ ] Fixtures typed with real domain types and matching the true emitted schema
- [ ] Shared fixtures extracted into typed factory helpers with `overrides`
- [ ] Hooks tested via `renderHook` + `act`; timer-driven logic uses fake timers
- [ ] Edge, error, and disabled/guard branches covered
- [ ] Tests colocated, grouped in `describe`, isolated via `beforeEach`/`afterEach`
